    Firmware overview

    Available firmwares

    What is firmware?

    Firmware is the program that runs inside of the Lynepad. It is responsible for detecting key-presses, and communicating with the connected computer. Different firmwares can behave differently, and support communication with a different software on your computer. You can replace the firmware by uploading (flashing) another.

    New default firmware

    The QMK firmware has become a default firmware starting from order #00007 on lynepad.com store.

    QMK is an open-source universal firmware for keyboards.
    Learn more about QMK in their docs.

    Pros

    • Supports Windows, MacOS, and Linux
    • Can be configured using VIA, a web-based configurator. No need to do any scripting or coding.
    • Supports 4 layers (key-binding layouts)
    • Doesn't require any client app (like AHK) running in the background
    • Community support on QMK and VIA Discord
    • Can be set and used later on another device (config is written into the Lynepad)

    Cons

    • More difficult AHK support out of the box
    • Color of the RGB Arc cannot be changed as it is used as layer indicator. You can turn it off or change the brightness.
    • Some special features such as RGB Arc following the thumwheel are missing

    The QVEX firmware sends F22 keystroke together with A - U keystrokes by default. This can be detected by AHK client and translated to your desired actions. Or you can use the QVEX Desktop app (alpha version) to reconfigure the default keystrokes Lynepad is sending.

    Pros

    • Define unlimited number of layers (key-binding layouts)
    • Define separate layer for each app running on your PC (automatic switching using process name)
    • Supports complex content-aware macros

    Cons

    • Windows only
    • Config is not written into the Lynepad, the keys are just interpreted by the AHK
    • Requires running AHK client
    • Configuration done through script writing

    Pros

    • Setup Lynepad using nice graphical interface
    • Highest level of utility as this is a proprietery app
    • Easy firmware updates
    • Support basic macros
    • Define up to 4 layers
    • Can be set and used later on another device (config is written into the Lynepad)

    Cons

    • Windows only (MacOS and Linux support will be added in the future) Currentely in alpha testing on our Discord
    • Very limited capabilities as QVEX Desktop is currentely an alpha version

    Flashing (uploading) a different firmware

    Experimental QVEX Flasher

    Use at your own risk. Do not use it without previous experience with flashing avr109 compatible boards.

    We are working on QVEX Flasher, a web-based tool for reflashing the Lynepad firmware that works under chromium-based desktop browsers on all major operating systems.

    After reflashing the firmware, perform a factory reset as there might be some leftover data stored on the EEPROM memory.

    1. Download .hex file of your desired firmware (QMK or QVEX Firmware)
    2. Download and run the QMK Toolbox.
    3. In case you haven't already done so when installing the QMK Toolbox, install the drivers by pressing Ctrl + N .
    4. Click the "Open" button and locate your downloaded firmware file.
    5. Plug in your Lynepad
    6. Check the auto-flash checkbox
    7. Reset the Lynepad with the provided SIM tool. The firmware should start uploading.

    2. Plug in your Lynepad
    3. Run the script below
    4. Reset the device to using provided SIM tool while the script is running

    #!/bin/bash

# Check if the .hex file argument is provided
if [ -z "$1" ]; then
    echo "Please provide the path to the .hex file as an argument."
    echo "Example: ./fwupload.sh my-hex-file.hex"
    exit 1
fi

# Extract the .hex file path from the first argument
hex_file="$1"

# Set the initial state flag
initial_state="true"

# Continuously check for newly connected USB devices
while true; do
    # Get the list of available devices
    devices=$(ls /dev/tty*)

    # Check if any new device is present
    if [[ $devices != "$old_devices" && $initial_state != "true" ]]; then
        # Iterate through the devices to find the target device
        while read -r device; do
            if [[ $device =~ "ttyACM" ]]; then
                # Upload the .hex file to the device using avrdude
                avrdude -p atmega32u4 -c avr109 -P "$device" -U flash:w:"$hex_file"

                # Exit the loop after successful upload
                break
            fi
        done <<< "$devices"
    fi

    # Save the current list of devices
    old_devices="$devices"

    # Set the initial state flag to false after the first run
    initial_state="false"

    # Wait for a few seconds before checking again
    sleep 5
done
